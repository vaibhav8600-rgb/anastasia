# M0.1 ‚Äî pywebview.api ‚Üí WS parity checklist (Phase 0, commit 5)

The frontend used to call `pywebview.api.<method>()` in-process. In the split
build it calls the same `call("<method>", ‚Ä¶)`, but `transport.js` routes it:

* **WS** ‚Äî a protocol `request` frame to anna-core; the reply resolves the
  promise. (These are exactly `REQUEST_METHODS` in `app/core/server.py`;
  `test_arg_spec_ws_methods_match_the_server_whitelist` fails CI if the two
  ever diverge.)
* **approval** ‚Äî `confirm` alone becomes an `approval` frame naming the card id
  (commit 3), never a generic request.
* **native** ‚Äî window-local only (a native file dialog, the bootstrap); served
  by anna-ui's tiny `NativeApi`, never crosses the socket.

**How to run this pass:** start `python app\main.py --core`, then
`python app\main.py --ui`. Do each *Trigger* below and confirm *Expect*. Every
row must behave exactly as it did in the single-process build ‚Äî that is the
whole point of M0.1. Then do the reconnect rows at the bottom.

Events flow the other way unchanged: core's `ui.dispatch(type, payload)` is sent
as an `event` frame and `transport.js` unwraps it back into
`window.ui.dispatch({type, payload})`, so all 30 event handlers in `app.js`
(`state_change`, `confirm_request`, `live_streaming`, `camera_capture`,
`full_state`, ‚Ä¶) render exactly as before. One new event, `connection`, drives
the reconnect banner and input gating; it does not exist in legacy mode.

### State indicators (added after the Phase-1 sanity pass)

The original checklist verified the request *routing* but not that the status
word reflects reality during a turn. It must ‚Äî the status line is how you trust
what the mic is doing, so it may never lie:

| # | Do this | Expect |
|---|---|---|
| E1 | Ask her a question that gets a **spoken** reply | status shows **Speaking** for the whole reply ‚Äî never "Ready"/"Listening" while audio is playing |
| E2 | Same, in **continuous/hands-free** mode | still **Speaking** while she talks; only after she finishes does it go to Listening (mic reopens) |
| E3 | **Barge in** (PTT while she speaks) | flips to **Listening** immediately (TTS cancelled, mic genuinely open) |
| E4 | During a **Gemini Live** session | reads **Listening (Live)** when idle, flips to **Speaking** while she talks (native Live audio bypasses SpeechOutput, so it's driven off the model's audio stream), back to Listening when her turn drains; never a bare "Ready" |

Priority is **speaking > listening > ready**, enforced in `Controller.set_state`
(`test_state_priority.py`). Note the mic during her speech is display-gated
either way ‚Äî the recorder's callback drops frames while `audio_gate.speaking` is
set (half-duplex), so "Speaking" is the honest word even if hands-free left the
device open.

## Reconnect rows (WS build only ‚Äî the honesty contract)

| # | Do this | Expect |
|---|---|---|
| R1 | Kill `--core`, watch the window | amber "Reconnecting to Anna‚Ä¶" banner; send/mic/toggles disabled |
| R2 | Click "Run it" on a card **while core is down** | card is **not** greyed; shows "Couldn't reach Anna‚Ä¶"; nothing runs |
| R3 | Restart `--core` | banner clears; UI re-hydrates from `full_state`; a pending card reappears **by id** |
| R4 | Approve that re-hydrated card | it runs now ‚Äî the same card, resolved once |
| R5 | Drop core mid-command, restart | conversation is intact and **not duplicated** (the snapshot is authoritative) |

The server halves of R2‚ÄìR5 are pinned in `tests/test_phase0_ui_client.py`; the
transport halves in `tests/js/transport.test.cjs`. This manual pass is the
human confirmation that the two meet correctly in a real WebView.
